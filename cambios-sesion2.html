Correcciones de entorno

📦
Dependencias de mapas no instaladas Fix
frontend/package.json → npm install

Al hacer git pull del trabajo del compañero, el frontend no arrancaba con estos errores: 

ERROR: Can't resolve 'leaflet/dist/leaflet.css'
ERROR: Can't resolve 'react-leaflet'

El compañero había añadido leaflet@1.9.4 y react-leaflet@5.0.0 al package.json, pero al hacer el pull no se ejecutó npm install, así que los paquetes no estaban en node_modules.

✅ Solución: ejecutar npm install dentro de la carpeta frontend/. El compilador descargó las dos dependencias y el frontend arrancó sin problemas.

Regla a recordar: siempre que hagas pull en un proyecto Node, ejecuta npm install por si alguien añadió paquetes nuevos.

🗄️
Backend no arrancaba — MySQL apagado Fix
XAMPP → MySQL

El backend se paraba nada más arrancar. El log mostraba centenares de líneas de error, pero la causa raíz era una sola: 

Caused by: java.net.ConnectException: Connection refused: getsockopt
→ HikariPool-1 - Starting... (fallo)
→ Communications link failure al puerto 3306

MySQL estaba parado. Spring Boot intenta conectarse a la BD nada más arrancar, y al no encontrar nada en el puerto 3306 lanza una cascada de errores que tapa el problema real.

⚠️ Todo el stacktrace de Hibernate, JPA y JWT son consecuencias en cadena. El único error real es que MySQL no estaba corriendo.
✅ Solución: abrir XAMPP y arrancar MySQL antes de lanzar el backend.

🔑
Error al registrarse — validación de contraseña Fix (diagnóstico)
Backend · UsuarioCreateDTO · @Pattern

Al intentar registrarse con la contraseña 12345678A el backend respondía con un 400. El log mostraba: 

MethodArgumentNotValidException:
  Field 'password': rejected value [12345678A]
  Pattern: ^(?=.*[a-z])(?=.*[A-Z])(?=.*\d).+$
  → La contraseña debe contener al menos una mayúscula,
    una minúscula y un número

La contraseña 12345678A tiene dígitos y una mayúscula, pero le falta al menos una minúscula. La regex del backend requiere los tres tipos de carácter.

✅ Solución: usar una contraseña como 12345678Aa (minúscula + mayúscula + número). No se modificó código; es un requisito intencionado de seguridad.
💡 Mejora pendiente: mostrar en el formulario los requisitos de la contraseña antes de enviar, para que el usuario sepa qué se le pide sin tener que adivinar.

Adicionalmente, el log mostraba un warning de "token válido pero usuario no existe en BD" — era un token antiguo guardado en localStorage de una sesión anterior con la BD vacía. Se resuelve borrando el localStorage desde DevTools → Application → Local Storage.

Modal "Ver Perfil" del profesional Feature
Profesionales.js · api/profesional.js

Cuando un cliente está viendo los profesionales de una subcategoría (p. ej. "Reparación de aire acondicionado"), cada tarjeta tiene dos botones: Ver Perfil y Contratar. El botón "Ver Perfil" estaba pintado pero sin funcionalidad. Se ha implementado completamente.

Cliente pulsa«Ver Perfil»
→
Se abre el modalcon skeleton
→
GET /profesionales/{id}fetch al backend
→
Modal completocon todos los datos

¿Qué muestra el modal?

  • Foto de perfil (o iniciales si no hay foto), nombre y ciudad
  • Valoración media con estrellas y número de opiniones
  • Bio / descripción profesional
  • Años de experiencia y nombre de empresa (si los tiene)
  • Código postal
  • Tarjeta del servicio concreto: título, descripción, precio/hora y duración

Comportamiento:

  • En móvil: aparece como bottom-sheet desde abajo
  • En desktop: modal centrado con overlay difuminado
  • Mientras carga el perfil del backend: muestra un skeleton animado
  • Si el fetch falla: el modal igual se abre con los datos del servicio (fail gracefully)
  • Bloquea el scroll del fondo mientras está abierto

Archivos modificados:

Código añadido en api/profesional.js:

// Obtiene el perfil completo de un profesional por su ID
export async function obtenerProfesionalPorId(id) {
  const res = await apiFetch(`/profesionales/${id}`);
  if (!res.ok) throw new Error("No se pudo cargar el perfil del profesional");
  return res.json();
}

Datos que devuelve el backend (ProfesionalDTO):

{
  id, descripcion, experiencia, nombreEmpresa,
  cif, plan, codigoPostal,
  valoracionMedia, numeroValoraciones, usuarioId
}

Estado añadido en el componente principal:

const [servicioSeleccionado, setServicioSeleccionado] = useState(null);
const [datosProfesional, setDatosProfesional]         = useState(null);
const [cargandoPerfil, setCargandoPerfil]             = useState(false);

Función que abre el modal y hace el fetch:

async function handleVerPerfil(servicio) {
  setServicioSeleccionado(servicio);  // abre el modal inmediatamente
  setDatosProfesional(null);
  setCargandoPerfil(true);
  try {
    const datos = await obtenerProfesionalPorId(servicio.profesionalId);
    setDatosProfesional(datos);
  } catch {
    // si falla, el modal igual se muestra con los datos del servicio
  } finally {
    setCargandoPerfil(false);
  }
}

Eliminado botón "Contratar" del modal de perfil Refinamiento UX
frontend/src/pages/public/profesionales/Profesionales.js · ModalPerfilProfesional

El modal "Ver Perfil" incluía en su pie un botón "Contratar este servicio", pero ese botón ya existe en la tarjeta principal del servicio. Tenerlo duplicado dentro del modal generaba redundancia y confusión.

Se eliminó el bloque <div> del pie del modal que contenía el botón: 

// Eliminado de ModalPerfilProfesional
<div className="border-t border-slate-100 px-6 py-4">
  <button className="w-full rounded-full bg-slate-900 ...">
    Contratar este servicio
  </button>
</div>

El botón "Contratar" sigue disponible en la TarjetaProfesional original, que es el punto natural desde el que el cliente puede iniciar la contratación.
