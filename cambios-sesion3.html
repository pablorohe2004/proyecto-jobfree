ReservaDTO — campos enriquecidos Backend
dto/reserva/ReservaDTO.java · ReservaMapper.java

El DTO original solo devolvía IDs numéricos (clienteId, servicioId). Para que el frontend pueda mostrar información legible sin hacer llamadas adicionales, se añadieron cuatro campos nuevos:

Campo añadidoValorQuién lo consume
clienteNombrenombre + apellidos del clientePanel del profesional (Solicitudes)
servicioTitulotítulo del servicio reservadoAmbos paneles
profesionalIdID del usuario profesionalUso interno / futuro

El mapper los rellena en un solo lugar (ReservaMapper.toDTO()) para que cualquier endpoint que devuelva un ReservaDTO los incluya automáticamente.

dto.setClienteNombre(r.getCliente().getNombre() + " " + r.getCliente().getApellidos());
dto.setServicioTitulo(r.getServicio().getTitulo());
dto.setProfesionalId(r.getServicio().getProfesional().getUsuario().getId());
dto.setProfesionalNombre(r.getServicio().getProfesional().getUsuario().getNombre() + " "
        + r.getServicio().getProfesional().getUsuario().getApellidos());

ReservaRepository — dos queries nuevas Backend
repository/ReservaRepository.java

El repositorio solo tenía una query: existsByServicioAndFechaInicio(). Se añadieron dos más para poder listar reservas por rol:

// Reservas donde el usuario es el cliente (para GET /reservas/mias)
List<Reserva> findByClienteId(Long clienteId);

// Reservas de servicios del profesional (para GET /reservas/recibidas)
List<Reserva> findByServicio_Profesional_UsuarioId(Long usuarioId);

Spring Data JPA genera el SQL automáticamente a partir del nombre del método, navegando las relaciones: Reserva → servicio → profesional → usuario → id.

ReservaService — notificaciones y corrección de bug Fix Backend
service/ReservaService.java

Bug corregido: confirmarReserva() enviaba la notificación de "reserva confirmada" al profesional en lugar de al cliente. Es decir, el profesional que acababa de aceptar recibía una notificación de sí mismo. Corregido para notificar al cliente:

// Antes (incorrecto):
Usuario profesional = guardada.getServicio().getProfesional().getUsuario();
notificacionService.crear("Tu reserva ha sido confirmada...", profesional);

// Después (correcto):
Usuario cliente = guardada.getCliente();
notificacionService.crear("Tu solicitud #" + guardada.getId() + " ha sido aceptada. Servicio: "
        + guardada.getServicio().getTitulo(), cliente);

Funcionalidad añadida: crearReserva() ahora notifica al profesional cuando llega una nueva solicitud:

// Al final de crearReserva(), tras el save:
Usuario profesional = nueva.getServicio().getProfesional().getUsuario();
notificacionService.crear(
    "Nueva solicitud para tu servicio: " + nueva.getServicio().getTitulo() +
    ". Revisa tus solicitudes para aceptarla o denegarla.",
    profesional);

Métodos de listado añadidos:

public List<Reserva> listarPorCliente(Long clienteId) {
    return reservaRepository.findByClienteId(clienteId);
}

public List<Reserva> listarPorProfesional(Long usuarioId) {
    return reservaRepository.findByServicio_Profesional_UsuarioId(usuarioId);
}

ReservaController — endpoints nuevos y seguridad reforzada Backend
controller/ReservaController.java

Se añadieron dos endpoints nuevos y se reforzó la seguridad del endpoint de confirmación:

EndpointRolDescripción
GET /reservas/miasAutenticadoDevuelve las reservas del cliente autenticado
GET /reservas/recibidasPROFESIONALDevuelve las solicitudes recibidas por el profesional
PATCH /reservas/{id}/confirmarPROFESIONAL ✦Ahora exige que sea el profesional del servicio concreto

✦ Antes cualquier usuario autenticado podía confirmar cualquier reserva que le "perteneciera". Ahora se verifica explícitamente que el usuario sea el profesional del servicio reservado.

@PreAuthorize("hasRole('PROFESIONAL')")
@PatchMapping("/{id}/confirmar")
public ResponseEntity<ReservaDTO> confirmarReserva(@PathVariable Long id) {
    Usuario usuario = getUsuarioAutenticado();
    Reserva reserva = reservaService.obtenerPorIdSeguro(id, usuario);
    Long profesionalUsuarioId = reserva.getServicio().getProfesional().getUsuario().getId();
    if (!usuario.getId().equals(profesionalUsuarioId)) {
        throw new ReservaInvalidaException("Solo el profesional del servicio puede aceptar esta solicitud");
    }
    return ResponseEntity.ok(ReservaMapper.toDTO(reservaService.confirmarReserva(reserva)));
}

ModalContratar — selección de fecha y hora Feature
Profesionales.js · componente ModalContratar

Al pulsar el botón "Contratar" en una tarjeta de servicio, se abre este modal. Muestra un resumen del servicio (nombre del profesional y precio) y un selector de fecha y hora nativo (<input type="datetime-local">).

  • La fecha mínima es la hora actual + 1 hora (no se pueden pedir servicios en el pasado)
  • Al pulsar "Enviar solicitud" llama a crearReserva() con el servicioId y la fecha elegida
  • Si hay error del backend (servicio inactivo, fecha ocupada…) lo muestra en el mismo modal
  • En éxito: cierra el modal y actualiza el botón de la tarjeta a "Pendiente"
  • El modal tiene overlay con backdrop-blur, bloquea el scroll del fondo y funciona como bottom-sheet en móvil
async function handleEnviar(e) {
  e.preventDefault();
  const reserva = await crearReserva(servicio.id, fecha + ":00");
  onExito(reserva); // cierra modal + actualiza botón a "Pendiente"
}

Solicitudes.js — panel del profesional Feature
pages/dashboard/profesional/Solicitudes.js · archivo nuevo

Nueva página del dashboard del profesional, accesible desde el menú lateral en Solicitudes. Muestra todas las solicitudes de contratación recibidas, organizadas en tres pestañas:

  • Pendientes — con contador en rojo si hay alguna sin atender
  • Aceptadas — historial de las que se aceptaron
  • Denegadas — historial de las que se rechazaron

Cada tarjeta de solicitud pendiente muestra: nombre del cliente, servicio solicitado, fecha y hora, precio estimado, y botones Aceptar (verde) y Denegar (rojo). Al pulsar, el estado de la tarjeta se actualiza en tiempo real sin recargar.

Al aceptar, el backend llama a PATCH /reservas/{id}/confirmar, que cambia el estado a CONFIRMADA y manda una notificación automática al cliente.

Bug fix — eliminar servicio con reservas existentes Fix
ReservaRepository.java · ServicioOfrecidoService.java

Problema: al pulsar "Eliminar" en un servicio que tenía al menos una reserva asociada, el backend lanzaba un error de clave foránea (Cannot delete or update a parent row: a foreign key constraint fails). La tabla reserva tiene una columna servicio_id con NOT NULL, y MySQL no permite borrar un servicio_ofrecido si hay filas que lo referencian.

Por qué ocurrió: la relación @OneToMany en ServicioOfrecido no tiene cascade = CascadeType.REMOVE, por lo que JPA no propaga el borrado a las reservas hijas automáticamente.

Solución: antes de borrar el servicio, se eliminan todas sus reservas usando un nuevo método de repositorio:

// ReservaRepository.java — nuevo método derivado
List<Reserva> findByServicioId(Long servicioId);

// ServicioOfrecidoService.java — eliminarServicio()
reservaRepository.deleteAll(reservaRepository.findByServicioId(id));
servicioRepository.delete(servicio);

Se prefirió esta opción frente a añadir cascade en la entidad porque es más explícita y evita borrados involuntarios si en el futuro se añaden relaciones adicionales sobre Reserva (pagos, valoraciones, mensajes).
